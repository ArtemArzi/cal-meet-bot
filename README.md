# 🗓️ Bot Vstrechi (Бот для подтверждения встреч)

![Python Version](https://img.shields.io/badge/python-3.12%2B-blue.svg)
![FastAPI](https://img.shields.io/badge/FastAPI-005571?style=flat&logo=fastapi)
![SQLite](https://img.shields.io/badge/sqlite-%2307405e.svg?style=flat&logo=sqlite&logoColor=white)
![Architecture](https://img.shields.io/badge/Architecture-Clean%2FHexagonal-success.svg)

Telegram-бот, который берет на себя всю рутину по подтверждению рабочих встреч. 

**Google Календарь — единственный источник истины.** Вы создаете встречу в календаре, а бот сам связывается с участниками в Telegram, собирает их подтверждения и автоматически обновляет статусы в календаре. Если кто-то не подтвердил встречу вовремя — она отменяется.

---

## ✨ Ключевые возможности

- **Синхронизация с Google Календарем:** Бот не создает встречи. Он слушает ваш календарь (через Service Account с Domain-Wide Delegation) и работает только с тем, что вы запланировали.
- **Умные дедлайны:**
  - *Обычная встреча:* Дедлайн подтверждения — 18:00 предыдущего рабочего дня.
  - *Встреча сегодня:* Дедлайн — за 2 часа до начала.
  - *Срочная встреча (Fast Track):* Дедлайн — за 10 минут до начала (если встреча создана менее чем за час).
- **Железобетонная надежность (Outbox Pattern):** Никаких потерянных сообщений. Все уведомления для Telegram и запросы к Google API сначала сохраняются в локальную базу данных, а затем гарантированно доставляются фоновыми воркерами.
- **Отсутствие раздутой базы данных (No ORM):** В проекте используется только чистый SQL (SQLite в режиме WAL) и нативные датаклассы Python. Максимальная производительность и предсказуемость.

## 🛠 Архитектура и Структура проекта

Бот спроектирован по принципам **Чистой Архитектуры (Hexagonal Architecture)**. Бизнес-логика полностью изолирована от базы данных, Telegram и Google API.

```text
src/bot_vstrechi/
├── domain/         # Чистая бизнес-логика (правила, state machine, модели)
├── application/    # Слой оркестрации (координация домена и инфраструктуры)
├── infrastructure/ # Низкоуровневые адаптеры (настройки, логирование)
├── db/             # Слой работы с БД (Raw SQL, Repository, Outbox)
├── telegram/       # Адаптер для Telegram API и формирование сообщений
├── calendar/       # Адаптер для Google Calendar API
├── workers/        # Фоновые процессы (Outbox, CalendarSync, Scheduler)
└── api/            # FastAPI вебхуки
```

## 🤖 Команды бота

Бот спроектирован так, чтобы минимизировать ручной ввод. Основное взаимодействие происходит через inline-кнопки (подтвердить/отменить).

| Команда | Описание | Доступ |
|---|---|---|
| `/start` | Начать работу с ботом и привязать свой Telegram ID к системе. | Все пользователи |
| `/help` | Справка по работе бота и правилам дедлайнов. | Все пользователи |
| `/chat` | Настроить чат для групповых статус-уведомлений. | Только Менеджеры (только ЛС) |
| `/people` | Управление пользователями и их привязкой (Telegram ↔ Email). | Только Менеджеры |

> **⚠️ Важно:** У бота **НЕТ** команд `/meet` или `/schedule`. Он не умеет создавать или редактировать время встреч. Любые изменения времени или состава участников должны делаться **только в Google Календаре**.

## 🚀 Как это работает (Workflow)

1. **Планирование:** Инициатор создает встречу в Google Календаре и добавляет участников.
2. **Синхронизация:** Фоновый воркер (`CalendarSyncWorker`) замечает новую встречу и переводит ее в статус `pending`.
3. **Опрос:** Бот отправляет участникам в Telegram кнопки: `[Подтвердить]` / `[Не смогу]`.
4. **Решение участников:** 
   - Если все подтвердили — встреча получает статус `confirmed`. Бот красит событие в календаре в зеленый цвет.
   - Если кто-то отказался — встреча переходит к инициатору для принятия решения (отменить или провести без участника).
5. **Дедлайны:** Если участник молчит, бот присылает напоминания. Если время вышло — встреча автоматически отменяется или требует ручного вмешательства создателя.

## ⚙️ Установка и запуск

### 1. Подготовка Google Workspace (DWD)
Для работы с календарями корпоративного домена боту нужен сервисный аккаунт с правами **Domain-Wide Delegation**.
1. Создайте сервисный аккаунт в Google Cloud Console.
2. Сгенерируйте и скачайте JSON-ключ.
3. В админке Google Workspace (`admin.google.com`) перейдите в *Security > API Controls > Domain-wide Delegation*.
4. Добавьте Client ID вашего сервисного аккаунта и укажите scope: `https://www.googleapis.com/auth/calendar.events`.

### 2. Переменные окружения (.env)
Создайте файл `.env` в корне проекта по образцу:
```env
# Телеграм
TELEGRAM_BOT_TOKEN="123456789:ABCdefGHIjklMNOpqrSTUvwxYZ"
TELEGRAM_WEBHOOK_URL="https://your-domain.com/webhook"

# База данных
DATABASE_URL="sqlite:///./var/local.db"

# Google Calendar
GOOGLE_SERVICE_ACCOUNT_FILE="/path/to/your/service-account.json"
GOOGLE_DELEGATED_ADMIN_EMAIL="admin@your-domain.com"
```

### 3. Установка зависимостей
```bash
git clone https://github.com/your-username/bot-vstrechi.git
cd bot-vstrechi
python -m venv .venv
source .venv/bin/activate
pip install -e .
```

### 4. Запуск приложения (Development)
В одном терминале запустите API:
```bash
uvicorn src.bot_vstrechi.asgi:app --reload
```
В отдельном терминале запустите фоновый цикл воркеров:
```bash
PYTHONPATH=src python3 -m bot_vstrechi.worker_entrypoint
```

### 5. Deployment (Production)
Для продакшена рекомендуется использовать **systemd** или **supervisor** для управления процессами воркеров. API должно работать за reverse proxy (Nginx/Traefik) и управляться через `gunicorn` с `uvicorn` воркерами.

## 📝 Разработка и Конвенции

Если вы хотите внести вклад в код, пожалуйста, ознакомьтесь с [Базой знаний проекта (AGENTS.md)](AGENTS.md). 
Ключевые правила:
- Никаких ORM и Pydantic в доменном слое.
- Изменения состояния базы данных и добавление задач в Outbox должны происходить в рамках **одной атомарной транзакции** (`self.atomic()`).
- В качестве идентификатора пользователя всегда используется `telegram_user_id` (int), а не `@username`.
